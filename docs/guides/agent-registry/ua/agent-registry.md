# Реєстр агентів (Admin Agent Registry)

Реєстр агентів — це центральний каталог платформи для управління AI-агентами. Він надає веб-інтерфейс для виявлення, встановлення, налаштування, увімкнення, вимкнення та моніторингу агентів, зареєстрованих на платформі.

## Модель даних

Агенти зберігаються в таблиці `agent_registry` з такими ключовими колонками:

| Колонка | Тип | Опис |
|---------|-----|------|
| `name` | `varchar` | Унікальний ідентифікатор агента (slug, наприклад `hello-agent`) |
| `version` | `varchar` | Семантична версія з Agent Card |
| `manifest` | `jsonb` | Повний JSON Agent Card (name, version, description, skills тощо) |
| `config` | `jsonb` | Конфіг, що редагується адміном (description, system_prompt) |
| `enabled` | `boolean` | Чи активний агент для маршрутизації подій/команд |
| `installed_at` | `timestamp` | Встановлюється при провіжинінгу; NULL = стан маркетплейсу |
| `health_status` | `varchar` | `healthy`, `degraded`, `unavailable`, `error`, `unknown` |
| `violations` | `jsonb` | Повідомлення про порушення конвенцій з discovery |
| `tenant_id` | `uuid` | Ізоляція тенанта |

Аудит-події зберігаються в `agent_registry_audit` з колонками: `agent_name`, `action`, `actor`, `payload`, `tenant_id`, `created_at`.

## Lifecycle агента

```
[Маркетплейс]  →  встановити  →  [Встановлено/Вимкнено]  →  увімкнути  →  [Встановлено/Увімкнено]
                                          ↑                                         ↓
                                       видалити                                  вимкнути
                                          ↑                                         ↓
                                  [Маркетплейс]          ←←←←←←←←←←←←←←←←←←←←←←←←
```

### Стани

| Стан | `installed_at` | `enabled` | Опис |
|------|---------------|-----------|------|
| Маркетплейс | `NULL` | `false` | Виявлено, але не провіжинено |
| Встановлено/Вимкнено | `IS NOT NULL` | `false` | Провіжинено, не маршрутизує події |
| Встановлено/Увімкнено | `IS NOT NULL` | `true` | Активний, маршрутизує події та команди |

### Дії lifecycle

- **Реєстрація**: Агент самостійно реєструється через `POST /api/v1/internal/agents/register` зі своїм Agent Card. Потребує заголовка `X-Platform-Internal-Token`.
- **Встановлення**: Адмін провіжинить сховища (Postgres/Redis/OpenSearch згідно маніфесту) та встановлює `installed_at`. Аудит-дія: `installed`.
- **Увімкнення**: Адмін активує агента для маршрутизації подій. Аудит-дія: `enabled`.
- **Вимкнення**: Адмін деактивує агента. Аудит-дія: `disabled`.
- **Видалення (Деінсталяція)**: Адмін депровіжинить сховища та очищає `installed_at`. Агент повертається до маркетплейсу. Аудит-дія: `uninstalled`.

## Адмін-інтерфейс

### Сторінка списку агентів (`/admin/agents`)

Сторінка списку агентів відображає всіх зареєстрованих агентів у двох вкладках:

- **Встановлені** — агенти з `installed_at IS NOT NULL`
- **Маркетплейс** — агенти з `installed_at IS NULL`

Кожен рядок показує: назву, версію, опис, badge статусу, badge здоров'я, дату оновлення та кнопки дій.

**Доступні дії за вкладками:**

| Вкладка | Дії |
|---------|-----|
| Встановлені (увімкнено) | Вимкнути, Налаштування |
| Встановлені (вимкнено) | Увімкнути, Видалити |
| Маркетплейс | Встановити |

Кнопка **Виявити агентів** запускає pull-based discovery (Traefik або Kubernetes) та оновлює реєстр.

### Сторінка налаштувань агента (`/admin/agents/{name}/settings`)

Сторінка налаштувань надає:

1. **Форма конфігурації** — поля `description` та `system_prompt`, що редагуються, зберігаються через `PUT /api/v1/internal/agents/{name}/config`
2. **Agent Card** — відображення версії, опису, команд, подій, skills, capabilities (тільки читання)
3. **Iframe адмін-панелі** — вбудована адмін-панель агента (якщо маніфест агента містить `admin_url`)

### Модальне вікно порушень

Агенти з порушеннями конвенцій показують badge **Degraded**. Натискання на badge відкриває модальне вікно зі списком повідомлень про порушення (наприклад, "Missing /health endpoint", "Missing Docker label").

## API-ендпоінти

Всі ендпоінти потребують `ROLE_ADMIN` (сесійна автентифікація), крім реєстрації, яка використовує `X-Platform-Internal-Token`.

| Метод | Шлях | Опис |
|-------|------|------|
| `POST` | `/api/v1/internal/agents/register` | Реєстрація/оновлення агента (token auth) |
| `GET` | `/api/v1/internal/agents` | Список всіх агентів (admin auth) |
| `POST` | `/api/v1/internal/agents/{name}/install` | Встановити агента |
| `POST` | `/api/v1/internal/agents/{name}/enable` | Увімкнути агента |
| `POST` | `/api/v1/internal/agents/{name}/disable` | Вимкнути агента |
| `PUT` | `/api/v1/internal/agents/{name}/config` | Зберегти конфіг агента |
| `DELETE` | `/api/v1/internal/agents/{name}` | Деінсталювати агента |

## Health Polling

Платформа запускає periodичний health poller (`AgentHealthPollerCommand`), який:

1. Виконує `GET /health` для кожного зареєстрованого агента
2. Оновлює `health_status` у реєстрі
3. Запускає перевірку конвенцій (перевіряє Docker-мітки, поля маніфесту, доступність ендпоінтів)
4. Зберігає порушення в колонці `violations`
5. Очищає застарілих агентів маркетплейсу (ніколи не встановлені, `health_check_failures >= 5`)

Inline health probe також запускається при реєстрації, якщо маніфест містить поле `health_url`.

## Discovery

Агенти виявляються через pull-based провайдери:

- **Traefik** — запитує Traefik API для сервісів з Docker-міткою `ai.platform.agent=true`
- **Kubernetes** — запитує Kubernetes API для подів з тією ж міткою

Discovery отримує Agent Card з `http://{hostname}/api/v1/manifest` та виконує upsert агента в реєстр.

## i18n

Всі рядки адмін-інтерфейсу використовують ключі перекладу Symfony через Twig-фільтр `|trans`. Файли перекладів знаходяться за адресою:

- `src/translations/messages.uk.yaml` — Українська (канонічна)
- `src/translations/messages.en.yaml` — Англійська

Ключі реєстру агентів мають префікс `agents.` (сторінка списку) та `agent_settings.` (сторінка налаштувань).
