# News Maker Agent

News Maker Agent відповідає за сканування джерел новин, витягування статей, їх ранжування та переписування у вигляді українськомовних резюме для AI-спільноти.

## Архітектура

Агент складається з кількох ключових компонентів:

1. **Crawler** - Отримує та витягує статті з налаштованих джерел новин
2. **Ranker** - Використовує LLM для оцінки та відбору найбільш релевантних статей
3. **Rewriter** - Перетворює обрані статті в українськомовні резюме
4. **Digest Generator** - Компілює кілька статей у цілісні дайджест-пости
5. **Admin Interface** - Веб-інтерфейс для налаштування та ручних запусків

## Функція глибокого сканування

### Огляд

Функція глибокого сканування дозволяє агенту знаходити статті не лише з базової URL джерел новин, але й з підсторінок, на які посилається головна сторінка. Це значно збільшує виявлення статей без необхідності реєстрації додаткових джерел.

### Налаштування

Глибоке сканування контролюється двома основними параметрами, доступними в адмін-інтерфейсі:

- **Глибина Crawl** (за замовчуванням: 1)
  - `1`: Сканувати лише базову URL (оригінальна поведінка)
  - `2`: Сканувати базову URL + один рівень підсторінок

- **Лінків на рівень** (за замовчуванням: 10)
  - Максимальна кількість посилань для переходу на кожному рівні глибини
  - Контролює широту сканування для запобігання надмірним HTTP-запитам

### Як це працює

1. **Глибина 0**: Отримати базову URL джерела та витягти до N посилань
2. **Глибина 1**: Для кожного посилання глибини-0, отримати сторінку та:
   - Витягти вміст статті, якщо сторінка є статтею
   - Витягти додаткові посилання для подальшого сканування (якщо глибина < max_depth)
3. **Дедуплікація**: URL, знайдені на кількох глибинах, отримуються лише один раз
4. **Обмеження домену**: Переходи здійснюються лише по посиланнях того ж домену або піддомену
5. **Тайм-бокси**: Тайм-аути на рівні джерела та запуску запобігають безконтрольному скануванню

### Схема бази даних

Функція глибокого сканування додає відстеження походження до сирих новинних елементів:

- `crawl_depth` (INTEGER): Рівень глибини, на якому була знайдена стаття (0 = базова URL, 1 = підсторінка)
- `discovered_from_url` (VARCHAR): URL сторінки, де було знайдено посилання

### Міркування щодо продуктивності

- **HTTP-запити**: На глибині 2 з 10 посиланнями на рівень, найгірший випадок ~111 запитів на джерело
- **Тайм-бокси**: Тайм-бокс джерела (за замовчуванням 240с) та тайм-бокс запуску (900с) запобігають надмірному скануванню
- **Моніторинг запитів**: Лічильники HTTP-запитів по глибинах логуються для моніторингу

### Опції налаштування

#### Змінні середовища (config.py)
```python
crawl_max_depth: int = 1                    # Глибина сканування за замовчуванням
crawl_max_links_per_depth: int = 10         # Посилань на рівень за замовчуванням
crawl_source_timebox_seconds: int = 240     # Тайм-аут на джерело
crawl_run_timebox_seconds: int = 900        # Загальний тайм-аут запуску
```

#### Налаштування бази даних (AgentSettings)
Конфігурація під час виконання через адмін-інтерфейс перевизначає значення середовища за замовчуванням:
- `crawl_max_depth`
- `crawl_max_links_per_depth`

### Міграція

Функція була додана через міграцію Alembic `003_add_crawl_depth_columns.py`:
- Додає nullable колонки зі значеннями за замовчуванням для зворотної сумісності
- Не потребує заповнення даних
- Існуюча поведінка зберігається при `max_depth=1`

### Моніторинг та логування

Crawler логує детальну статистику для кожного джерела:
- Посилання, знайдені на кожній глибині
- HTTP-запити, зроблені на рівень глибини
- Кількість доданих/існуючих/невдалих статей
- Статистика фільтрації (зовнішні, статичні, заблоковані тощо)

Приклад виводу логу:
```
Source 'TechCrunch' depth=1 produced 8 candidate links (href=45 accepted=8 filtered=37 offsite=12 static=15 blocked=3 duplicate=7 invalid=0)
Source 'TechCrunch' done: added=3 existing=2 fetch_failed=1 extract_failed=2 requests_by_depth={0: 1, 1: 8, 2: 15}
```

### Тестування

Функція включає комплексні тести, що покривають:
- Зворотну сумісність з `max_depth=1`
- Багаторівневе сканування з mock-даними
- Забезпечення обмеження глибини
- Обмеження домену через глибини
- Дедуплікацію між глибинами

### Стратегія відкату

Для відключення глибокого сканування:
1. Встановити `crawl_max_depth=1` в налаштуваннях адміна
2. Це негайно відновлює оригінальну поведінку одного рівня
3. Колонки бази даних можуть бути видалені в майбутній міграції за потреби
